# CI/CD × コンテナ 発表スクリプト(5分)

## スライド1: タイトル(20秒)

本日は「CI/CDとコンテナ」について、私が研修で実際に構築した環境を例にお話しします。
一言で言うと「git pushするだけで、数分後に本番環境が新しくなる仕組み」の話です。

## スライド2: コンテナとは(60秒)

まずコンテナです。

コンテナは、アプリ本体だけでなく、Javaやライブラリ、設定ファイルといった実行環境まで丸ごと1つの「イメージ」に箱詰めする技術です。

これの何が嬉しいかというと、「自分のPCでは動いたのに、サーバーだと動かない」という問題が原理的に起きなくなります。箱の中身はどこで動かしても同じだからです。

イメージの設計図がDockerfileで、ここに「ベースとなるOSイメージ、アプリの配置、起動コマンド」を書きます。

AWSでは、作ったイメージを保管する倉庫がECR、イメージを実際に動かす実行基盤がECS Fargateです。Fargateはサーバー管理が不要で、CPUとメモリを指定するだけでAWSが実行環境を用意してくれます。

## スライド3: CI/CDとは(60秒)

次にCI/CDです。

CIは継続的インテグレーション。コードをpushするたびに、ビルドとテストを自動で実行して、壊れていたらすぐ検知する考え方です。

CDは継続的デリバリー。ビルドした成果物を、自動でリリースまで運ぶことです。

本質は「人がやると忘れる、間違える手順を、機械に毎回まったく同じ手順でやらせる」ことです。手順書を見ながらの手作業デプロイをなくします。

AWSでは、Gitリポジトリを起点に、パイプライン全体を管理するCodePipeline、ビルド作業を担当するCodeBuild、pushを検知して起動するEventBridge、という役割分担で実現します。

## スライド4: 実際の構成(90秒)

ここからは、研修で構築したRevivalBookStoreNeoでの実例です。Spring Bootで作った書店アプリを、この仕組みでAWSに載せました。

流れはこの1本です。

まず、開発者がCodeCommitのリポジトリにgit pushします。

すると、EventBridgeがそのpushイベントを検知して、CodePipelineを起動します。

パイプラインの中では、CodeBuildがbuildspec.ymlに書かれた手順に従って、Dockerイメージをビルドし、ECRにプッシュします。

最後に、ECS Fargateが新しいイメージでコンテナを起動し、古いコンテナと順番に入れ替えます。これをローリングデプロイと呼び、サービスを止めずに更新できます。

1つ補足すると、○○ではDockerが使えないのですが、ビルドはすべてCodeBuildがクラウド側で実行するため、ローカルにDockerがなくても問題ありませんでした。これもこの構成の利点だと感じています。

## スライド5: 効果(60秒)

効果は3つあります。

1つ目、ミスがなくなります。デプロイ手順が完全に固定化されるので、「ファイルの置き忘れ」や「手順の飛ばし」が構造的に起きません。

2つ目、速くなります。pushから数分で反映されるので、リリース作業に人の時間を使いません。

3つ目、戻せます。ECRにはイメージがタグつきで残るので、問題が起きたら前のバージョンのイメージに切り替えるだけでロールバックできます。

## スライド6: まとめ(30秒)

まとめです。

コンテナが「どこでも同じに動く箱」を作り、CI/CDが「箱を作って入れ替える作業」を自動化する。この2つの組み合わせで、リリースの品質を人の注意力ではなく、仕組みで保証できるようになります。

自動化の価値は楽をすることではなく、ミスを仕組みで防ぐことだと学びました。

以上です。ありがとうございました。
